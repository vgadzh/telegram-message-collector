# Telegram Message Collector

Production-oriented Go service for connecting Telegram user accounts, receiving incoming messages, filtering them according to business rules, and storing relevant messages in PostgreSQL.

The project is designed as a portfolio and learning project demonstrating:

- Domain-Driven Design (DDD)
- Hexagonal Architecture
- Distributed Runtime Management
- Kubernetes Deployment
- Observability
- Production-Ready Go Practices

Current status: **Architecture & Foundation Phase**

---

# Project Overview

The service allows platform users to connect one or more Telegram accounts and continuously receive messages from selected senders.

Main goals:

- authenticate Telegram user accounts
- maintain long-lived Telegram sessions
- restore sessions after restart
- process Telegram updates reliably
- store filtered messages in PostgreSQL
- run multiple service replicas without duplicate processing

The database is the source of truth for all session state.

```text
HTTP API
    |
    v
Application Layer
    |
    +--------------------+
    |                    |
    v                    v
Session Manager    Runtime Reconciler
                         |
                         v
                  Telegram Runtime
                         |
                         v
                     PostgreSQL
```

Key architectural principles:

- DDD-inspired design
- Hexagonal architecture
- Database as source of truth
- Runtime reconciliation
- Explicit ownership of active sessions
- Graceful shutdown
- Context propagation
- Minimal dependencies

# Session Lifecycle

Authorization flow:

```text
NEW
    |
    v
SENDING_CODE
    |
    v
CODE_REQUIRED
    |
    +----> PASSWORD_REQUIRED
    |
    v
AUTHORIZED
    |
    v
ACTIVE
```

Failure states:

```text
FAILED
DELETED
```

A FAILED session cannot be restarted and must be recreated.

---

# Runtime Reconciliation

HTTP requests never start runtime components directly.

Instead, API requests modify desired state in PostgreSQL.

A background reconciler continuously compares:

```text
Desired State (Database)
vs
Actual State (Runtime)
```

and applies required actions:

```text
ACTIVE + not running
    -> start subscriber

INACTIVE + running
    -> stop subscriber
```

This approach guarantees recovery after restart and simplifies distributed deployment.

# Distributed Ownership

Multiple service replicas are supported.

A Telegram session may be owned by only one node at a time.

Ownership is coordinated through PostgreSQL leases.

Example:

```text
session #42
owner_node = api-1
lease_expires_at = ...
```

If a node disappears, another node may acquire ownership after lease expiration.

This prevents:

- duplicate processing
- session corruption
- update state races

# How To Run

Kubernetes manifests are the primary deployment target. Docker Compose is provided only for local development and quick evaluation.

Requirements:

- Go 1.24+
- PostgreSQL 17+
- Telegram API credentials

Before first start set ADMIN_PASSWORD_HASH.

## Local development

```bash
make up
make run
```

## Kubernetes

Environment variables:

```text
APP_ENV=local

POSTGRES_DSN=

TELEGRAM_API_ID=
TELEGRAM_API_HASH=

JWT_SECRET=
```

Start infrastructure:

```bash
make up
```

Run migrations:

```bash
make migrate
```

Start service:

```bash
make run
```
